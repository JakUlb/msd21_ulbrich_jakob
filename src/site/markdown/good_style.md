mv#How to write good source code comments
+ Always keep documentation in line with source code changes, if you change one change the other
+ Do not write comments describing what a certain part of your code does, let the code speek for itself. Instead write why you chose to do it like this.
+ what problem was being solved
+ who provided the code
+ why the solution is recommended 
+ what commenters thought of it 
+ whether it still works 
+ how it can be improved

#How to write good commit comments
+ What was done where.
+ structure a long comment in Header - linebreak - text
+ short summary in heading should have less than 50 chars and no "." at the end
+ line breaks every 72 chars
+ only describe what and why not how
